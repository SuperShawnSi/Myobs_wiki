# Transformer 通俗课

## 这篇课解决什么问题

很多人会背一句：

- Transformer 很重要
- 大模型都跟 Transformer 有关
- 核心是 Attention

但如果继续问：
- 它到底解决了什么旧问题？
- 为什么它比 RNN/LSTM 强？
- Self-Attention 到底在干嘛？
- 它和上下文长度、RAG、成本有什么关系？

很多人就开始发虚。

这篇课的目标不是让你推公式，而是让你达到下面这个程度：

1. 你能用人话解释 Transformer。
2. 你知道它为什么成为大模型底层架构。
3. 你知道它的优点和代价。
4. 你能把它和产品里的真实问题连起来。

## 一、先从 Transformer 之前的世界讲起

在 Transformer 之前，做文本任务最常见的是：
- RNN
- LSTM
- GRU

你现在不需要把这些模型学透，但要知道它们有一个共同特点：

**它们通常按顺序一个词一个词往后读。**

比如一句话：

```text
I went to Sydney because it was beautiful.
```

传统序列模型的处理方式更像这样：

```text
看到 I
  -> 再看到 went
  -> 再看到 to
  -> 再看到 Sydney
  -> 再看到 because
  -> 再看到 it
  -> 再看到 was
  -> 再看到 beautiful
```

### 这种方式有什么问题

#### 1. 很难并行
因为前一个词没处理完，后一个词很难完全开始。

#### 2. 长距离依赖难学
如果句子很长，前面的信息要一路传到后面，容易丢。

#### 3. 文本一长，训练和推理都更吃力
尤其在长文档、复杂上下文场景下，旧模型很容易表现不稳定。

这也是为什么后来大家会一直寻找一种更适合处理长文本、更适合大规模训练的架构。

## 二、Transformer 到底想解决什么

Transformer 的核心想法可以先粗暴记成一句话：

**处理当前词时，不再老老实实一格一格往前传，而是直接去看整段里和自己最相关的部分。**

这就是 Attention，特别是 Self-Attention（自注意力）的直觉。

## 三、什么是 token

在讲 Attention 前，先把最基础的词讲清楚。

模型不直接理解“词”或“字”的语义，它先把文本切成一个个更小单位，这些单位叫：

**token**

### 例子

一句英文：

```text
Transformer is powerful.
```

可能会被切成：

```text
["Transformer", " is", " powerful", "."]
```

一句中文：

```text
今天天气很好。
```

也可能被切成：

```text
["今天", "天气", "很", "好", "。"]
```

注意：
- token 不一定等于一个字
- 也不一定等于一个单词
- 它是模型内部处理文本的最小单位

所以以后你看到：
- token 数量
- 上下文长度
- token 成本

都要知道它说的是模型实际在处理的文本碎片数。

## 四、什么是 embedding

模型看不懂文字本身，它要先把 token 变成数字向量。

这个过程通常叫：

**embedding（嵌入）**

你可以把它理解成：

- 把“词”变成一串坐标
- 让模型能在数学空间里处理它们之间的关系

### 粗暴理解

假设：

```text
"cat" -> [0.2, 0.8, -0.1, ...]
"dog" -> [0.1, 0.7, -0.2, ...]
"car" -> [-0.9, 0.1, 0.4, ...]
```

如果训练得好，语义接近的词，在向量空间里也更接近。

## 五、如果只有 embedding，还不够

Transformer 有一个天然问题：

**它不像 RNN 那样天生按顺序走。**

这意味着如果你只把所有 token 扔给模型，它并不知道谁在前、谁在后。

比如这两句：

```text
狗咬人
人咬狗
```

token 可能都差不多，但顺序完全不同，意思也完全不同。

所以 Transformer 还需要让模型知道：

- 第 1 个 token 在哪
- 第 2 个 token 在哪
- 第 3 个 token 在哪

这个信息通常通过：

**位置编码（positional encoding）**

来提供。

## 六、Self-Attention 到底在干什么

这是核心中的核心。

### 一句话版本

当模型在处理某个 token 时，它会去看整段文本里其他 token，并决定：

- 谁更重要
- 谁更相关
- 我应该更关注谁

### 一个最经典的理解例子

句子：

```text
The animal didn't cross the street because it was tired.
```

当模型看到 `it` 时，它需要判断 `it` 指的是什么。

- 是 `animal`？
- 还是 `street`？

如果只顺序传递信息，这件事会比较费劲。

但 Self-Attention 的做法是：

- 处理 `it` 这个词时
- 直接看前面所有词
- 给每个词一个“相关性分数”
- 最后更关注和 `it` 真正相关的那个词

### 更像人的理解方式

你读一句话时，也不是机械地一字一字死记。你会自然地在脑中回看和当前词最相关的部分。

Transformer 想做的，就是把这种“回看上下文”的能力，变成一个可训练的机制。

## 七、Self-Attention 的最小直觉图

假设一句话有 4 个 token：

```text
A B C D
```

处理 `C` 时，模型会做类似这样的事：

```text
C 看 A：相关性 0.1
C 看 B：相关性 0.7
C 看 C：相关性 0.2
C 看 D：相关性 0.0
```

最后 `C` 的表示，就会更多吸收 `B` 的信息。

这就是“关注”。

## 八、Query、Key、Value 是什么

如果你以后面试或阅读资料，几乎一定会看到这三个词：

- Query
- Key
- Value

你不需要推公式，但最好理解成一个粗暴比喻。

### 图书馆比喻

把一个 token 想成在图书馆里找资料的人。

#### Query
它当前想找什么。

#### Key
其他 token 提供的“标签”或“索引”。

#### Value
其他 token 真正能提供给你的信息内容。

### 过程
处理当前 token 时：
1. 先拿自己的 Query
2. 去和所有 token 的 Key 比一遍
3. 看谁更匹配
4. 然后按匹配程度，把那些 token 的 Value 加权汇总回来

这就是 Self-Attention 的基本直觉。

## 九、用伪代码理解 Attention

下面这段不是严格实现，而是帮助你建立脑内流程。

```text
输入：一句话的所有 token 表示

对于每个 token_i：
  计算 token_i 的 query
  让 token_i 去看所有 token_j：
    计算 query_i 和 key_j 的相关程度
  把所有相关程度做归一化
  用这些权重去加权汇总所有 value_j
  得到 token_i 的新表示
```

如果你想看更“像代码”的版本，可以这样理解：

```python
for token_i in tokens:
    scores = []
    for token_j in tokens:
        score = similarity(query(token_i), key(token_j))
        scores.append(score)

    weights = softmax(scores)
    token_i_new = weighted_sum(weights, [value(t) for t in tokens])
```

你不用会写这个实现，但要知道：

**Attention 本质上是在做“为当前 token 分配注意力，再汇总上下文信息”。**

## 十、什么是 Multi-Head Attention

你可以把它理解成：

**不是只用一个视角看上下文，而是同时用多个视角去看。**

### 为什么需要多个 head

一句话里，不同关系可能都重要：
- 语法关系
- 指代关系
- 情感关系
- 时间关系
- 实体关系

如果只有一个注意力头，它可能很难同时兼顾这些不同模式。

所以 Multi-Head Attention 就像：
- 一个头看语法
- 一个头看实体
- 一个头看指代
- 一个头看情感

当然这不是手动指定的，是模型训练出来的。

### 粗暴类比

你在开会时：
- 一个人专门看时间线
- 一个人专门看风险
- 一个人专门看预算
- 一个人专门看用户反馈

最后再把这些视角汇总。

## 十一、Transformer 的基础结构长什么样

最经典的原始论文结构分为：
- Encoder
- Decoder

### Encoder 更像什么
把输入理解清楚。

### Decoder 更像什么
根据已有输出和输入信息，逐步生成结果。

不过你现在更常碰到的是：
- 只做理解的 Encoder 类模型，例如 BERT
- 只做生成的 Decoder 类模型，例如 GPT

### 为什么 GPT 对你重要
你现在接触的大多数大模型产品，更多是 Decoder-only 路线。

因为它更适合：
- 续写
- 对话
- 生成
- 总结
- 改写

## 十二、为什么 Transformer 对大模型特别重要

它强在三件事：

### 1. 并行训练能力强
不像旧模型那样一步一步顺序计算，所以更适合大规模训练。

### 2. 更善于捕捉长距离依赖
远距离 token 可以直接互相“看见”。

### 3. 容易扩展
参数变大、数据变多时，效果提升更明显。

这就是为什么后来很多大模型都围绕 Transformer 系列发展。

## 十三、Transformer 的代价是什么

你不能只记它强，还要记它贵。

### 1. 上下文越长，算力压力越大
因为 Self-Attention 需要比较 token 和 token 之间的关系。

简单理解：
- token 越多
- 两两关系越多
- 成本和延迟越容易上升

### 2. 不是“上下文越长越好”
上下文长，确实能看更多信息。
但也会带来：
- 更高延迟
- 更高成本
- 更多无关噪音

### 3. 它擅长建模上下文，不等于天然知道最新事实
这就是为什么后来还需要：
- RAG
- 工具调用
- 外部知识源

## 十四、把 Transformer 和产品问题连起来

这部分对你最重要。

## 1. 为什么 Prompt 结构很重要
因为 Transformer 很依赖上下文。

如果你给模型的输入：
- 目标不清楚
- 规则不清楚
- 输出格式不清楚

那模型就可能抓错重点。

所以 Prompt 工程本质上是在帮 Transformer：
- 更好地理解你到底要什么
- 更好地关注正确的信息

## 2. 为什么上下文窗口是产品问题
面试官如果问你“上下文窗口为什么重要”，你要知道不是在问你背参数，而是在问：
- 长文档能不能放进去
- 多轮对话能不能记住前文
- 成本和延迟会不会失控

## 3. 为什么 RAG 会和 Transformer 配套出现
因为 Transformer 能处理上下文，但知识不是无限新、无限准。

所以很多产品做法是：
- 先把外部知识检索出来
- 再把检索结果放进 Transformer 的上下文里
- 让模型基于这些信息生成回答

## 十五、一个和你项目相关的例子

### `COMP9900`
如果没有 Transformer 这种上下文建模能力，大模型很难：
- 看懂一段长反馈
- 归纳多个评论的共性
- 把不同评论和不同主题联系起来

### `PP@SS AI 元数据助手`
它需要看：
- 图片内容
- EXIF
- 表单草稿
- 你给它的输出规则

本质上也是让模型在多种输入上下文中抓重点。

## 十六、初学者最容易误解的 5 件事

### 误解 1：Transformer 就是 ChatGPT
不对。

Transformer 是一种架构思想。
ChatGPT 是基于这一类架构和后续训练方法做出来的产品。

### 误解 2：Attention 就是“理解力”
不完全对。

Attention 是一种机制，帮助模型分配关注点。它不是“像人一样理解”的同义词。

### 误解 3：上下文越长一定越好
不对。

上下文长会增加可参考信息，但也会增加成本、延迟和噪音。

### 误解 4：Transformer 解决了一切知识问题
不对。

它仍然会：
- 知识过时
- 幻觉
- 误读输入

所以你后面还得学 RAG、工具调用和评估。

### 误解 5：AI PM 不需要懂 Transformer
也不对。

你不需要会写底层实现，但你必须知道：
- 模型为什么擅长看上下文
- 为什么长文本和上下文窗口重要
- 为什么成本和延迟会跟 token 数挂钩

## 十七、你在面试里怎么回答“Transformer 是什么”

### 简洁版
```text
Transformer 是当前大模型最重要的基础架构之一，它的核心突破是 Self-Attention。也就是处理当前 token 时，模型可以直接去看整段文本里和自己最相关的其他 token，而不需要像早期 RNN/LSTM 那样一步一步顺序传递信息。这样它更容易并行训练，也更擅长处理长距离依赖，所以后来成为大模型的主流基础。
```

### 稍展开版
```text
我理解 Transformer 最重要的点有三个。第一，它通过 Self-Attention 让每个 token 都能直接关注上下文中更相关的部分；第二，它比早期序列模型更容易并行训练，所以更适合大规模模型；第三，它虽然很擅长上下文建模，但上下文越长，成本和延迟也会上升，所以在产品里要同时考虑效果、速度和成本，而不是只说模型很强。
```
