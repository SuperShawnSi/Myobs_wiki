# RAG 通俗课

## 这篇课解决什么问题

很多人会背一句：

- RAG 就是检索增强生成
- 先检索，再回答

但如果继续问：
- 为什么不直接问大模型？
- 检索到底在检什么？
- 什么叫 chunk、embedding、召回、重排？
- 为什么做了 RAG 还是会答错？
- 产品经理要管哪些环节？

很多人就开始发虚。

这篇课的目标是：

1. 让你真正理解 RAG 的完整链路。
2. 让你知道 RAG 解决什么，不解决什么。
3. 让你知道 AI 产品经理在 RAG 场景里要负责什么。
4. 让你能把 `COMP9900` 讲成一个更像 RAG 产品的案例。

## 一、RAG 到底是什么

RAG 全称是：

**Retrieval-Augmented Generation**

中文常说：

**检索增强生成**

一句话先记住：

**先查资料，再让模型基于查到的资料回答。**

它不是让模型“凭空更聪明”，而是给模型在回答前补一批外部知识。

## 二、为什么需要 RAG

如果你直接问大模型，它的问题通常有三类：

### 1. 知识可能过时
模型参数里的知识不是实时更新的。

比如你问：
- 公司最新制度
- 最新课程安排
- 刚刚更新的政策

模型未必知道。

### 2. 可能编造
即使它语气很自信，也不代表它有依据。

### 3. 很难给出处
你可能得到一个“看起来合理”的答案，但不知道它是从哪来的。

这就会在很多场景里很危险：
- 企业知识问答
- 教育反馈总结
- 医疗、法律、金融等高风险文本辅助

RAG 的核心价值就是：

**让回答更多依赖外部知识源，而不是只依赖模型参数记忆。**

## 三、RAG 最小链路长什么样

先给一个最小版本：

```text
1. 用户提出问题
2. 系统把问题转成便于检索的表示
3. 去知识库里找相关内容
4. 把找到的内容连同问题一起喂给模型
5. 模型基于这些内容生成答案
```

你可以把它想成：

- 大模型是一个“会写答案的人”
- 检索系统是一个“先去档案室找材料的人”

RAG 的意思就是：

**让写答案的人，先拿到资料再写。**

## 四、什么叫知识库

RAG 里的知识库不一定是一个复杂数据库。

它可以是：
- PDF 文档
- FAQ
- 课程反馈文本
- 内部制度文档
- 说明书
- 网页内容
- 结构化表格

关键不是形式，而是：

**这些内容是不是你希望模型在回答时参考的外部知识源。**

## 五、RAG 里的几个关键名词

## 1. Chunk（切片）
如果文档很长，不能整本都直接塞进模型。

所以通常要先把文档切成较小片段，这些片段叫：

**chunk**

### 为什么要切片
因为：
- 一整篇太长
- 检索粒度太粗
- 模型上下文有限

### 例子
一篇课程反馈总结说明文档，可能会切成：
- 第一段：课程目标相关
- 第二段：作业难度相关
- 第三段：讲师表现相关
- 第四段：实验环节相关

## 2. Embedding（向量化）
为了让系统能按语义找相似内容，常见做法是：

- 把每个 chunk 变成向量
- 把用户问题也变成向量
- 然后比较它们的相似度

这一步叫 embedding。

### 粗暴理解
不是按关键词一字不差去搜，而是按“语义像不像”去找。

## 3. Retrieval（检索）
检索就是从知识库里把相关 chunk 找出来。

常见两种：
- 关键词检索
- 向量检索

很多实际系统会混用。

## 4. Rerank（重排）
初步检索出来的内容，不一定排序最好。

所以有时还会再做一步：
- 对候选片段重新排序
- 把最相关的几段放前面

这叫重排。

## 5. Grounding / 引用依据
当系统回答时，如果能告诉用户：
- 这个答案主要来自哪几段内容
- 哪个结论对应哪条证据

那用户会更信任。

这就是很多 RAG 产品里常见的：
- 引用
- 来源卡片
- 证据展开

## 六、一个完整的 RAG 产品链路

如果你把 RAG 想完整一点，它往往是这样：

```text
离线阶段：
  收集知识源
    -> 清洗文本
    -> 切片 chunk
    -> 生成 embedding
    -> 建索引

在线阶段：
  用户提问
    -> 问题改写或标准化
    -> 检索相关 chunk
    -> 可选重排
    -> 组装 prompt
    -> 调模型生成答案
    -> 返回答案 + 引用依据
```

## 七、用伪代码理解 RAG

下面给你一个“产品经理能看懂”的伪代码版本。

```python
def answer_question(user_question):
    # 第一步：把问题转成检索可用的表示
    query_vector = embed(user_question)

    # 第二步：从知识库里找最相关的片段
    candidate_chunks = vector_search(query_vector, top_k=5)

    # 第三步：把检索结果拼进提示词
    prompt = build_prompt(
        question=user_question,
        context=candidate_chunks
    )

    # 第四步：让模型生成答案
    answer = llm_generate(prompt)

    # 第五步：返回答案和引用依据
    return {
        "answer": answer,
        "citations": candidate_chunks
    }
```

如果不用向量，也可能是：

```python
candidate_chunks = keyword_search(user_question)
```

重点不在代码，而在你要知道：

**RAG 不是一个步骤，而是一整条链路。**

## 八、为什么做了 RAG 还是会错

这点非常重要。

很多人误以为：
- 只要上了 RAG
- 幻觉就消失了
- 答案就一定可靠了

这是错的。

RAG 会错，常见有 5 类原因。

### 1. 知识源本身有问题
如果文档过时、错漏、矛盾，检索再好也没用。

### 2. 切片切得不好
如果 chunk 太大：
- 噪音多
- 重点不集中

如果 chunk 太小：
- 上下文不完整
- 语义断裂

### 3. 检索没找准
问的是 A，找回来的是 B，看起来沾边，其实不对。

### 4. 找到了，但模型总结错了
这就是生成阶段的问题。

### 5. 展示方式让用户误解
即使系统拿到了依据，如果引用不清楚，用户仍然会误以为答案 100% 正确。

## 九、产品经理在 RAG 里要管什么

RAG 不是算法一个人的活。

AI 产品经理至少要盯下面这些点：

## 1. 这个场景到底需不需要 RAG
不是所有问题都要上 RAG。

适合 RAG 的通常是：
- 需要依赖外部知识
- 知识经常变化
- 用户需要看到依据
- 回答不能只靠模型常识

不一定适合 RAG 的包括：
- 创意写作
- 风格改写
- 简单分类
- 单纯生成文案

## 2. 知识源来自哪里
你必须知道：
- 哪些文档被纳入知识库
- 更新频率是什么
- 谁负责维护
- 哪些内容不能入库

## 3. 检索出来后怎么展示
用户不一定需要看到全部 chunk，但很多场景需要：
- 引用来源
- 证据展开
- “未找到依据”的提示

## 4. 错了怎么办
你必须设计：
- 无结果怎么办
- 低置信度怎么办
- 是否显示“请人工核实”
- 是否允许用户反馈“这条不对”

## 5. 怎么评估
至少分三层：
- 检索质量
- 生成质量
- 产品结果

## 十、结合 `COMP9900` 来理解 RAG

这是你最适合讲的例子。

### 1. 为什么它适合 RAG
因为它要处理的是：
- 大量课程反馈文本
- 老师关心的不是“一个漂亮总结”
- 而是“这个结论有没有依据、能不能追溯到具体评论”

这就天然适合：
- 检索相关评论
- 生成总结
- 同时保留证据引用

### 2. 为什么它不能只做普通摘要
如果只是普通摘要工具，问题会很大：
- 结论看起来像对，但不知道来自哪
- 小样本意见可能被夸大
- 老师无法验证总结是否可靠

所以你们才会强调：
- 证据追溯
- 结构化结果
- 小样本隐私提醒
- 异常和边界展示

### 3. 这就是 AIPM 的价值
在这个项目里，真正像 AIPM 的地方，不是“你会不会接 RAG”，而是：
- 你知道为什么这里必须有证据
- 你知道为什么不能把评论原文乱展示
- 你知道为什么需要结构和隐私边界

## 十一、RAG 和微调有什么区别

这也是高频问题。

### 微调更像什么
把模型“训练得更适合某类任务或语气”。

### RAG 更像什么
不改变模型本体，而是在回答前给它查资料。

### 一句话区别
- 微调：改模型本身的行为倾向
- RAG：给模型外部知识参考

### 为什么很多知识型产品先上 RAG
因为：
- 更新知识更方便
- 不需要每次改知识都重新训练模型
- 更容易保留来源和依据

## 十二、RAG 产品常见设计问题

### 1. 要不要展示引用
通常建议展示，尤其在知识密集场景。

### 2. 检索不到怎么办
不要硬答。应允许：
- “未找到足够依据”
- “请缩小问题范围”
- “目前知识库中没有相关信息”

### 3. 要不要让用户看原文
很多场景建议支持展开原文，因为这能增强信任和核验能力。

### 4. 是否一定要用向量库
不一定。小规模知识库用关键词也可能够用。关键是看：
- 数据规模
- 语义复杂度
- 召回要求

## 十三、你在面试里怎么回答“RAG 是什么”

### 简洁版
```text
RAG 就是检索增强生成，核心思想是先从外部知识库里找相关内容，再让模型基于这些内容生成答案。它适合知识密集型场景，因为能提升回答的时效性和可追溯性。但 RAG 不是银弹，效果还取决于知识源质量、切片方式、检索质量和结果呈现方式。
```

### 稍展开版
```text
我理解 RAG 不是简单地“给模型喂更多字”，而是把知识获取和答案生成拆成两步。第一步是从外部知识源里检索出和问题最相关的内容，第二步是让模型基于这些内容生成结果。它特别适合企业知识问答、文档问答、课程反馈洞察这类需要依据的场景。对产品来说，重点不只是把检索链路接上，还要考虑引用展示、低置信度提示、无结果兜底，以及最终怎么评估检索和生成是否真的有效。
```
